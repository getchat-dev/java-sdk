# Contributing

Notes for working on the GetChat Java SDK itself. If you only want to *use* the
SDK, the [README](README.md) is all you need — this file is for changing its
source.

## Build and test

```bash
./gradlew build     # compile, check formatting, static null analysis, test, jar, javadoc
./gradlew test      # tests only
./gradlew test --tests '*SignatureVectorTest*'   # just the cross-SDK signing conformance suite
./gradlew spotlessApply        # reformat the sources in place
./gradlew publishToMavenLocal  # install into ~/.m2 for a consuming app to try
```

Java 17 or newer. The build sets `options.release = 17` rather than pinning a
Gradle toolchain, so it runs on any JDK ≥ 17 you already have.

## Formatting

Sources are formatted with **palantir-java-format** driven by **Spotless**
(4-space indent, 120 columns — the style the project already used).
`spotlessCheck` runs as part of `build`, so unformatted code fails the build;
run `./gradlew spotlessApply` before committing. palantir is deliberately
non-configurable — it breaks fluent chains of three or more calls onto separate
lines. That is the house style now; run the formatter rather than hand-fighting
it.

## Static analysis

The main sources compile under **Error Prone + NullAway** (JSpecify mode), so the
`@NullMarked` / `@Nullable` contract is checked by the compiler, not just
documented — a wrong nullability annotation fails the build. Test sources are
exempt (they deliberately pass `null` to assert `NullPointerException`
behaviour). `EqualsContractTest` runs **EqualsVerifier** over the value types.

## The rules that matter

A few of these are load-bearing. Breaking one does not just fail a test — it can
put wrong bytes on the wire that the backend silently rejects, or diverge this
SDK from the other GetChat SDKs.

- **Tests are the spec — never edit a test to make it pass.** A red test is
  information about the code, not the test. The only time you change a test is a
  deliberate contract change that *is* the task, agreed up front, with the test
  edit as part of that change. Editing a test *after* seeing it fail is the
  forbidden direction.

- **The golden signature vectors are a contract, not a fixture.**
  `SignatureVectorTest` replays 23 URLs generated by the reference **node SDK**
  that this SDK must reproduce character for character
  (`src/test/resources/signature-vectors.json`). A failure here is **never**
  "just update the JSON" — it means the two SDKs disagree about what the backend
  will accept; find out which one is right first. Regenerate the fixture only
  after a deliberate protocol change, and only from a checkout of the node SDK
  (`getchat-dev/node-sdk`) sitting next to this one as `../node`:

  ```bash
  ../node/node_modules/.bin/tsx scripts/gen-vectors.mts > src/test/resources/signature-vectors.json
  ```

- **The signing layer is frozen on purpose.** Everything under
  `dev.getchat.sdk.internal` reproduces JavaScript / PHP semantics byte for byte
  because the bytes it produces feed a signature the backend recomputes
  independently. Code there is un-idiomatic Java by design — do not "clean it up".
  The `TransportTest` byte assertions and the golden vectors are what prove it
  stayed correct.

- **One runtime dependency: Jackson.** Java has no built-in JSON, so Jackson is
  allowed — declared `implementation`, never `api`, so no Jackson type appears on
  a public signature. JSpecify is annotations-only (a contract, not a library).
  Do not add further runtime dependencies (no OkHttp, Guava, Apache Commons).
  Test-only dependencies (JUnit, EqualsVerifier) are unconstrained.

- **Exception policy.** Every deliberate validation throws `GetChatException` (or
  one of its subtypes) — never `IllegalArgumentException` /
  `IllegalStateException`. A raw `NullPointerException` is reserved for a `null`
  passed where a required reference is mandatory. See the `Errors` section of the
  README for the subtype hierarchy.

- **Dependency versions live in `gradle/libs.versions.toml`**, not inline in
  `build.gradle.kts`. The reason a version is pinned goes next to it in the
  catalog; the reason a dependency is wired a certain way goes next to its usage
  in `build.gradle.kts`.

## Extending the REST layer

The authority on the real wire format is the OpenAPI spec of the reference
**node SDK**, [`getchat-dev/node-sdk`](https://github.com/getchat-dev/node-sdk),
kept honest by that SDK's live tests. (A dedicated documentation repository will
host the spec later; this pointer will move there when it does.) Add a wrapper by
building an `ApiRequest`, funnelling it through the private transport,
percent-encoding every path parameter, and pinning the method / path / body in a
new `TransportTest` case. Mirror the node SDK's input coercions so both SDKs put
the same bytes on the wire.
